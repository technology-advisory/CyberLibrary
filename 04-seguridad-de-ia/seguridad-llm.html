04.01 · Seguridad de IA

Seguridad LLM

Protege entradas, contexto, salidas, secretos y consumo del modelo. Esta guía combina una explicación entendible para perfiles no técnicos con controles y evidencias útiles para arquitectura, seguridad, cumplimiento y auditoría.

En pocas palabras

La seguridad de un LLM exige proteger instrucciones, contexto, entradas, salidas, secretos y conexiones externas. Los filtros de contenido son una capa más; no sustituyen autorización, aislamiento ni validación de acciones.

Cómo abordarlo paso a paso

Modelar flujos de confianza

Distingue instrucciones del sistema, usuario, RAG, herramientas y memoria.

Reducir datos sensibles

Minimiza contexto, redacta secretos y aplica controles antes de enviar al modelo.

Validar salidas

Trata la respuesta como datos no confiables antes de ejecutar, renderizar o almacenar.

Limitar capacidades

Separa lectura de escritura y exige autorización fuera del modelo.

Probar ataques realistas

Incluye prompt injection, fuga, evasión, abuso de coste y contenido peligroso.

Checklist práctica

  • Existe evidencia de: diagrama de flujo de contexto.
  • Existe evidencia de: catálogo de secretos prohibidos.
  • Existe evidencia de: resultados de red teaming.
  • Existe evidencia de: configuración de cuotas.

Evidencias que conviene conservar

  • Diagrama de flujo de contexto
  • Catálogo de secretos prohibidos
  • Resultados de red teaming
  • Configuración de cuotas

Conserva únicamente la evidencia necesaria, con acceso restringido, versión y fecha. La trazabilidad no debe crear un nuevo repositorio de datos sensibles.

Errores que suelen aparecer

  • Confiar en “ignora instrucciones maliciosas”
  • Insertar credenciales en prompts
  • Ejecutar texto generado sin validación

Indicadores útiles para este tema

  • Intentos de inyección detectados
  • Fugas bloqueadas
  • Coste por sesión
  • Salidas rechazadas por validación
