11.01 · Cadena de suministro de IA

Model SBOM y AI-BOM

Crea un inventario verificable de todos los componentes de un sistema de IA. Esta guía combina una explicación entendible para perfiles no técnicos con controles y evidencias útiles para arquitectura, seguridad, cumplimiento y auditoría.

En pocas palabras

Un AI-BOM amplía el SBOM para registrar modelos, datasets, prompts, adaptadores, librerías, herramientas, servicios y licencias. Su valor está en responder rápidamente qué sistemas dependen de un componente afectado.

Cómo abordarlo paso a paso

Definir unidad de inventario

Acordar qué componentes y relaciones se representarán.

Extraer desde pipelines

Genera información desde CI/CD, registries, notebooks y catálogos.

Relacionar dependencias

Vincula aplicación, modelo, dataset, runtime, tool y proveedor.

Añadir identidad e integridad

Incluye versión, hash, licencia, origen y responsable.

Usar el BOM operacionalmente

Conecta vulnerabilidades, retiradas y cambios con análisis de impacto.

Checklist práctica

  • Existe evidencia de: ai-bom versionado.
  • Existe evidencia de: hashes de artefactos.
  • Existe evidencia de: grafo de dependencias.
  • Existe evidencia de: historial de generación.

Evidencias que conviene conservar

  • AI-BOM versionado
  • Hashes de artefactos
  • Grafo de dependencias
  • Historial de generación

Conserva únicamente la evidencia necesaria, con acceso restringido, versión y fecha. La trazabilidad no debe crear un nuevo repositorio de datos sensibles.

Errores que suelen aparecer

  • Listar solo paquetes Python
  • No incluir modelos remotos o APIs
  • Generar el BOM y no usarlo en incidentes

Indicadores útiles para este tema

  • Sistemas con AI-BOM
  • Componentes sin versión
  • Tiempo para localizar dependencia
  • Cambios detectados automáticamente
